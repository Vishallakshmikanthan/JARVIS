> This file provides authoritative context and rules for GitHub Copilot when generating or editing code in this repository. Follow every section strictly.

---

## 1. Project Overview

JARVIS is a local-first, voice-enabled AI assistant with dual-persona support (JARVIS / FRIDAY). It runs entirely offline (LLM via Ollama, TTS via Kokoro-ONNX, face auth via OpenCV) with optional cloud skill integrations (weather, news, Spotify, DuckDuckGo).

**Primary goals:**
- Fast, responsive, offline-capable assistant
- Modular, skill-based extensibility
- Persona-driven UX with natural language interaction
- Persistent memory across sessions

## 2. Architecture

```
jarvis-assistant/
├── main.py               # Entry point. Dual-mode: voice loop or text REPL
├── config.py             # Single Config class; all constants and .env overrides
├── core/
│   ├── brain.py          # Ollama LLM wrapper; persona-aware prompts; memory logging
│   ├── router.py         # Hybrid intent classifier (LLM + keyword fallback)
│   ├── persona.py        # Persona dataclasses, PersonaManager singleton, response banks
│   └── memory.py         # SQLite-backed conversation, preferences, and facts storage
├── voice/
│   ├── listen.py         # Porcupine wake-word listener (daemon thread)
│   ├── stt.py            # Speech-to-text via SpeechRecognition + PyAudio
│   └── tts.py            # Kokoro-ONNX TTS, lazy-loading, async playback
├── auth/
│   ├── face.py           # OpenCV LBPH face authentication
│   └── train_face.py     # Face model training pipeline
├── skills/               # Self-contained skill modules (one function per intent)
├── gui/
│   └── interface.py      # Tkinter dark-themed GUI (animated orb, threaded callbacks)
├── assets/               # Static resources (wake-word models, face images)
├── logs/                 # Loguru log files
└── models/               # Cached ONNX / ML model files
```

### Data Flow

```
Voice → [listen.py: wake-word] → [stt.py: speech-to-text]
                                        ↓
Text input ─────────────────→ [router.py: classify intent]
                                        ↓
                        ┌───────────────┼───────────────┐
                  skill intent     general intent    persona intent
                        ↓               ↓               ↓
               [skills/*.py]     [brain.py: LLM]   [persona.py]
                        └───────────────┼───────────────┘
                                        ↓
                              [memory.py: log turn]
                                        ↓
                              [tts.py: speak / GUI: display]
```

## 3. Coding Rules

### General
- **Python 3.10+** with `from __future__ import annotations` at the top of every file.
- **Type-hint all function signatures** — parameters and return types.
- Use `dataclasses.dataclass` for structured data objects (not plain dicts or NamedTuples).
- Never use `print()` for runtime output — use `loguru.logger` exclusively.
- All public identifiers use `snake_case`. Classes use `PascalCase`.
- Keep functions **single-responsibility**. If a function exceeds ~40 lines, split it.
- No magic numbers or hardcoded strings — reference `config.py` constants.
- Optional third-party imports must be guarded with `_AVAILABLE` flag pattern (see §6).

### Imports
- Standard library first, then third-party, then local project imports.
- Separate each group with a blank line.
- Avoid wildcard imports (`from module import *`).

### Error Handling
- Use `try/except` with specific exception types — never bare `except:`.
- On recoverable errors: `logger.error(...)` + return a sensible fallback value.
- On fatal startup errors: `logger.critical(...)` then `sys.exit(1)`.
- Never swallow exceptions silently.

### Threading
- Use `threading.Thread(daemon=True)` for all background tasks.
- Never block the main thread or GUI event loop.
- Use `threading.Event` for inter-thread signaling (not `time.sleep` polling).
- Thread-safe GUI updates must use `root.after(0, callback)` — never call Tkinter directly from a worker thread.

### Config Access
- Always read settings from the `Config` class in `config.py`.
- Instantiate once per module at module level: `cfg = Config()`.
- Never read environment variables directly with `os.environ` outside `config.py`.

## 4. Skill Creation Pattern

Each skill lives in `skills/<name>.py` and must follow this exact structure:

```python
from __future__ import annotations

import logging
from loguru import logger

# Guard optional dependencies
try:
    import some_library
    _AVAILABLE = True
except ImportError:
    _AVAILABLE = False

from config import Config

cfg = Config()


def get_<skill_name>(parsed_input: str) -> str:
    """
    One-line summary of what this skill does.

    Args:
        parsed_input: The cleaned user query (trigger words already stripped).

    Returns:
        A human-readable response string suitable for TTS and display.
    """
    if not _AVAILABLE:
        return "Sorry, the required library for this skill is not installed."

    try:
        # Core skill logic here
        result = _fetch_data(parsed_input)
        logger.info(f"[skill:<skill_name>] result={result!r}")
        return result
    except Exception as exc:
        logger.error(f"[skill:<skill_name>] failed: {exc}")
        return "Sorry, I encountered an error with that request."


def _fetch_data(query: str) -> str:
    """Internal helper — not exported."""
    ...
```

**Rules for skills:**
- The entry-point function must be named `get_<skill>` or `handle_<skill>`.
- Register the skill in `core/router.py` intent map under the correct intent key.
- `parsed_input` is already stripped of trigger words — do not strip again.
- Return a plain sentence string, not rich objects.
- Skills must be stateless — no global mutable state inside skill modules.

## 5. Logging Standard

This project uses **loguru** exclusively. Never use `logging.getLogger`.

| Level | When to use |
|-------|-------------|
| `logger.debug` | Verbose internal state, intermediate values |
| `logger.info` | Normal operation milestones (skill called, intent resolved) |
| `logger.warning` | Recoverable unexpected state (missing optional config, fallback used) |
| `logger.error` | Caught exceptions, failed API calls, degraded responses |
| `logger.critical` | Unrecoverable startup failure |

**Log message format: `[module:function] context=value action`**

```python
# Good
logger.info(f"[router:classify] intent=weather confidence=0.92")
logger.error(f"[brain:think] ollama_error={exc}")

# Bad — vague, no context
logger.info("Done")
logger.error("Something went wrong")
```

Log files are written to `logs/` as configured in `config.py`. Do not hardcode log paths.

## 6. Function Format

Every function must follow this template:

```python
def function_name(param1: Type1, param2: Type2 = default) -> ReturnType:
    """
    One-line summary (imperative mood, e.g. "Classify intent from raw text.").

    Args:
        param1: Description of param1.
        param2: Description of param2. Defaults to `default`.

    Returns:
        Description of return value.

    Raises:
        ValueError: If param1 is empty.
    """
    if not param1:
        raise ValueError("param1 must not be empty")

    logger.debug(f"[module:function_name] param1={param1!r}")

    try:
        result = _do_work(param1, param2)
        logger.info(f"[module:function_name] completed result={result!r}")
        return result
    except SpecificError as exc:
        logger.error(f"[module:function_name] failed: {exc}")
        return fallback_value
```

**Optional dependency guard (use when imports may fail):**

```python
try:
    import optional_lib
    _OPTIONAL_AVAILABLE = True
except ImportError:
    _OPTIONAL_AVAILABLE = False
    logger.warning("[module] optional_lib not installed — feature disabled")
```

**Singleton module-level exports (follow existing core modules):**

```python
# At bottom of module file
module_singleton = ModuleClass()
```

## 7. Persona System

- All persona definitions live in `core/persona.py` as `Persona` dataclasses.
- To add a new persona: add a `Persona(...)` entry to the `PERSONAS` registry dict.
- Persona responses are randomized from banks — never hardcode a single response string.
- `PersonaManager` is a singleton — import it as `from core.persona import persona_manager`.
- The active persona dictates: TTS voice ID, system prompt injected to LLM, response phrasing.

---

## 8. Memory System

- Import as `from core.memory import memory`.
- Use `memory.log_conversation(role, content)` after every user/assistant turn.
- Use `memory.get_recent(n)` to retrieve the last `n` turns for LLM context.
- Use `memory.set_preference(key, value)` / `memory.get_preference(key)` for persistent settings.
- Never write to the SQLite database directly — always use the `Memory` API.

---

## 9. GUI Conventions

- All GUI code lives in `gui/interface.py`.
- The `JarvisGUI` class owns all Tkinter widgets; do not create widgets outside it.
- Long-running operations always run in `threading.Thread(daemon=True)`.
- Communicate results back to GUI via `root.after(0, callback)` — never cross-thread Tkinter calls.
- Use `set_processing_state(True/False)` to toggle the animated orb state.
- Use `append_log(sender, text)` to post any message to the conversation log — it is thread-safe.
